# 一、Gradle 插件相关错误

**❌ 错误 1：Plugin [id: 'io.spring.dependency-management', version: '1.2.4', apply: false] was not found in any of the following sources**

* 提示：借助AI想使用最新版本，更新后遇到的问题

**报错现象**

执行 Build Project / compileJava 时直接失败，提示插件找不到，无其他多余日志。

**报错原因**

* io.spring.dependency-management 插件截至 2026.01 最新稳定正式版仅为 1.1.6，SpringBoot4.0.x 官方文档标注的1.2.x为规划版本，暂未发布；
* Gradle 插件仓库中无该版本，导致拉取插件失败。

**解决方案**

移除`io.spring.dependency-management`，因为官网给出的[发布时间是2024年12月](https://plugins.gradle.org/plugin/io.spring.dependency-management)，
[springboot 4.0.1](https://spring.io/projects/spring-boot#support)发布于2025年11月

**必坑要点**
* `io.spring.dependency-management` 核心价值不是「帮你声明版本」，而是帮你实现了【版本的自动兜底 + 传递依赖强制统一】，这是插件唯一的核心，也是它和 Gradle 原生写法的本质区别
  * Gradle 原生语法中，ext{} 只是定义了变量，帮你「集中管理版本号、简化修改」，仅此而已；
  * io.spring.dependency-management 插件，除了帮你集中声明版本，还多了一个Maven 同款的核心能力：「声明一次版本，全局自动生效」，包括直接依赖 + 传递依赖
* 废话：不知道`io.spring.dependency-management`什么意思，AI给了上面描述，决定删除。后面麻烦的就是要根据maven确认相关版本。

# 二、依赖引入规范

## 规范 1：java 插件 vs java-library 分工（绝对不能乱）
```groovy
dependencies {
    plugins {
      // 纯库模块，固定用 java-library，业务模块改成java
      id 'java-library'
    }
    // api修饰这些依赖）
    api 'org.springdoc:springdoc-openapi-starter-webmvc-ui'
}
```
* ✅ java-library → mom-common、mom-infra 纯库模块，无启动类、被其他模块依赖，用api暴露依赖；
* ✅ java → mom-admin 业务模块，有启动类、可独立运行，打包成可执行 jar；

## 彻底删除所有子模块的「测试依赖」

* 根工程已经全局注入了 Junit5 所有依赖，子模块重复写会导致版本冲突 + 编译报错，这是最容易犯的错！

## 依赖版本的优先级规则

* 根工程ext声明版本 → 子模块省略版本号自动匹配；
* 根工程resolutionStrategy.force → 最高优先级，强制所有依赖用指定版本，哪怕子模块写了其他版本也会被覆盖；
* 子模块声明的依赖 → 只对当前模块生效，优先级低于根工程的 force 规则。

### Gradle 依赖核心关键字对比表（结合多模块项目实战）

|关键字类型|关键字|核心作用|依赖传递性|生效阶段|典型使用场景（你的项目）|注意事项|
|---|---|---|---|---|---|---|
|**依赖作用域**|`implementation`|模块自用依赖，控制依赖可见性|❌ 不可传递|编译期 + 运行期|1. infra 依赖 common 模块<br>2. infra 引入 druid、mybatis-plus<br>3. admin 引入 web-starter|95% 场景首选，避免依赖泛滥|
||`api`|模块对外暴露依赖，下游模块自动继承|✅ 可传递|编译期 + 运行期|common 模块引入 feign、springdoc（所有模块都要用）|仅公共库核心依赖使用，严控范围|
||`compileOnly`|仅编译期需要，打包时不打入 JAR|❌ 不可传递|仅编译期|Lombok 依赖（编译生成代码，运行时无需）|搭配 `annotationProcessor` 使用|
||`runtimeOnly`|仅运行期需要，编译期无需引用|❌ 不可传递|仅运行期|PostgreSQL/MySQL 数据库驱动（SPI 加载）|Spring 官方推荐驱动依赖写法|
||`testImplementation`|仅测试代码的编译+运行生效|❌ 不可传递|测试代码的编译+运行期|JUnit5 核心依赖（根工程全局声明）|子模块无需重复配置|
||`testCompileOnly`|仅测试代码编译期需要，打包不打入|❌ 不可传递|仅测试代码编译期|测试用 Lombok 依赖|搭配 `testAnnotationProcessor`|
||`providedRuntime`|运行期需要，由容器提供（如 Tomcat），打包不打入|❌ 不可传递|仅运行期（容器环境）|War 包部署时的 Tomcat 依赖|仅传统容器部署场景使用|
|**版本强制规则**|`force`（resolutionStrategy 内）|强制统一项目中依赖的唯一版本，无视其他地方的版本声明|全局生效|依赖解析阶段（最早）|1. 统一 SpringBoot/Cloud 版本<br>2. 锁定 mybatis-plus、druid 版本|优先级最高，子模块版本会被覆盖|
#### 补充：核心优先级规则（必背）

1. **版本优先级**（从高到低）：

`force 声明版本` > 子模块手动写的版本 > 传递依赖版本 > SpringBoot 插件默认版本

1. **依赖传递优先级**：

`api` 传递依赖 > `implementation` 不传递 > `compileOnly/runtimeOnly` 不传递

#### 你的项目「硬性使用规范」（表格版）

|模块|插件类型|关键字使用原则|
|---|---|---|
|mom-common|java-library|1. 核心公共依赖用 `api`（feign、springdoc）<br>2. 自用依赖用 `implementation`<br>3. Lombok 用 `compileOnly`|
|mom-infra|java-library|1. 依赖 common 用 `implementation`<br>2. 数据库驱动用 `runtimeOnly`<br>3. druid、mybatis-plus 用 `implementation`|
|mom-admin|java|1. 依赖 common/infra 用 `implementation`<br>2. 只引入业务 starter（web、actuator）<br>3. 不重复引入基建依赖|
|根工程|全局配置|1. `ext` 集中声明所有版本<br>2. `force` 强制统一版本<br>3. 全局声明 Lombok、JUnit5 依赖|
