# 项目DDD架构总览

## 1. 项目简介

本项目采用领域驱动设计(DDD)架构，将系统划分为多个业务域模块，每个模块独立负责特定的业务功能。

## 2. 模块划分

| 模块名称 | 业务功能 | 主要领域对象 |
|---------|---------|------------|
| mom-common | 公共工具库 | SnowflakeIdGenerator, StringUtils |
| mom-infra | 基础设施支持 | 数据库配置, 连接池配置 |
| mom-iot | 物联网设备管理 | 设备(Device), 设备状态(DeviceStatus) |
| mom-admin | 系统管理 | 用户(User), 角色(Role), 权限(Permission) |
| mom-mes | 制造执行系统 | 工单(WorkOrder), 生产计划(ProductionPlan) |
| mom-wms | 仓库管理系统 | 仓库(Warehouse), 库存(Inventory), 物料(Material) |
| mom-qms | 质量管理系统 | 质量检验(QualityInspection), 质量问题(QualityIssue) |
| mom-gateway | API网关 | 路由配置, 认证授权 |

## 3. DDD分层架构

所有业务域模块(mom-iot, mom-admin, mom-mes, mom-wms, mom-qms)均采用统一的DDD分层架构：

```
模块目录/src/main/java/com/ly/factmesh/模块名称/
├── Application.java                     # Spring Boot应用入口
├── domain/                              # 领域层：核心业务逻辑
│   ├── entity/                          # 实体：具有唯一标识的业务对象
│   ├── valueobject/                     # 值对象：无唯一标识的不可变对象
│   ├── aggregate/                       # 聚合根：管理聚合边界的对象
│   ├── service/                         # 领域服务：跨实体的业务逻辑
│   ├── repository/                      # 仓储接口：聚合根持久化抽象
│   └── event/                           # 领域事件：领域内的重要事件
├── application/                         # 应用层：协调领域层和基础设施
│   ├── service/                         # 应用服务：处理应用用例
│   ├── dto/                             # 数据传输对象：层间数据传递
│   ├── command/                         # 命令对象：表示要执行的操作
│   └── query/                           # 查询对象：表示查询条件
├── infrastructure/                      # 基础设施层：技术实现
│   ├── repository/                      # 仓储实现：实现仓储接口
│   └── database/                        # 数据库相关配置
└── presentation/                        # 表示层：用户交互
    └── controller/                      # API控制器：处理HTTP请求
```

- 统计分析：质量数据统计与分析

## 5. 跨模块通信

各业务域模块之间通过以下方式进行通信：

1. **领域事件**：通过事件总线发布和订阅领域事件
2. **REST API**：通过API网关调用其他模块的REST接口
3. **服务注册与发现**：通过Nacos实现服务的动态注册与发现

## 7. ID生成策略

所有模块统一使用Snowflake算法生成ID，确保ID的全局唯一性、有序性和可扩展性。

```java
// Snowflake ID结构：1位符号位 + 41位时间戳 + 10位工作机器ID + 12位序列号
// 优点：全局唯一、含时间戳、可排序、无主键冲突
// 适用：分布式部署、高并发写入、跨库关联
```

## 8. 数据库设计原则

1. **每个业务域模块有独立的数据库**
2. **使用BIGINT类型存储Snowflake ID**
3. **外键约束仅用于同一模块内的表关联**
4. **跨模块关联使用ID引用**
5. **使用合适的索引优化查询性能**

## 9. 架构设计原则

1. **领域优先**：核心业务逻辑封装在领域层
2. **分层清晰**：严格遵循DDD分层架构
3. **模块独立**：每个业务域模块独立开发、部署
4. **接口驱动**：通过接口定义模块间的通信契约
5. **事件驱动**：通过领域事件实现模块间的松耦合通信
6. **可测试性**：设计便于单元测试和集成测试的代码
